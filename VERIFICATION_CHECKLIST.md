# 技術ニュース検証チェックリスト

## 概要
このチェックリストは、技術ニュースの収集・記載における品質管理を強化し、情報の正確性を担保するための詳細な検証手順を提供します。

## 📋 基本検証チェックリスト

### ✅ 情報源の検証（必須）
- [ ] **引用元リンクの設置**: すべてのニュース項目に引用元URLが含まれている
- [ ] **リンクの動作確認**: 引用元URLが正常にアクセス可能である
- [ ] **一次ソースの確認**: 可能な限り公式発表や一次ソースを引用している
- [ ] **信頼性の評価**: 情報源が技術コミュニティで信頼されている

### ✅ 内容の正確性検証（最重要）
- [ ] **リンク先内容との照合**: 記載内容が引用元の情報と完全に一致している
- [ ] **技術的詳細の確認**: バージョン番号、リリース日、機能名等が正確である
- [ ] **事実関係の確認**: 発表内容、企業名、製品名等に誤りがない
- [ ] **日付情報の確認**: ニュースの発表日時が正確に記載されている

### ✅ 形式・表現の確認
- [ ] **Markdownフォーマット**: 指定されたMarkdown形式で記載されている
- [ ] **セクション分類**: 適切な優先度・カテゴリに分類されている
- [ ] **簡潔性**: タイトルと概要が簡潔で理解しやすい
- [ ] **中立性**: 個人的意見ではなく客観的事実を記載している

## 🔍 詳細検証手順

### 1. 情報収集段階での検証

#### 情報源の評価基準
**推奨レベル A（最優先で使用）**
- 技術の公式ブログ・ドキュメント
- GitHub公式リリースページ
- 標準化団体の公式発表
- 主要企業の公式技術ブログ

**推奨レベル B（注意深く使用）**
- 著名な技術メディア（確認要）
- 開発者コミュニティの公式発表
- 技術カンファレンスの公式情報

**推奨レベル C（避けるべき）**
- 個人ブログ（著名な専門家以外）
- 未確認の情報・噂
- 翻訳記事（元ソース不明）

#### 情報収集時の確認項目
```markdown
□ ソースの信頼性レベル: [A/B/C]
□ 発表日時: [YYYY-MM-DD]
□ 情報の種類: [公式発表/ベータ情報/噂/分析記事]
□ 確認済み情報源数: [件数]
```

### 2. 記載内容の検証

#### 必須確認項目
- **製品名・技術名**: 正式名称の確認
- **バージョン情報**: 正確なバージョン番号
- **リリース状況**: ベータ/RC/正式版の区別
- **重要な機能**: 主要な新機能・変更点
- **影響範囲**: 誰に・何に影響するか

#### 記載例の検証
**❌ 不適切な例:**
```markdown
- **React新バージョン** - いくつかの新機能が追加
  [📖 どこかで見た記事](非公式URL)
```

**✅ 適切な例:**
```markdown
- **React 19.1.0がリリース** - Server Componentsの安定化とConcurrent Renderingの改善
  [📖 React 19.1 Release Notes](https://react.dev/blog/2025/06/05/react-19-1)
```

### 3. 品質スコアリング

#### 各項目の評価基準（3段階）

**情報源の信頼性**
- 3点: 公式ソース（レベルA）
- 2点: 準公式ソース（レベルB）  
- 1点: 非公式ソース（レベルC）

**内容の正確性**
- 3点: 引用元と完全一致、複数ソースで確認済み
- 2点: 引用元と一致、追加確認あり
- 1点: 引用元との照合のみ

**記載の完全性**
- 3点: 必要情報すべて記載（バージョン、日付、詳細等）
- 2点: 主要情報記載、一部詳細不足
- 1点: 最低限の情報のみ

**品質スコア計算**: (情報源 + 正確性 + 完全性) / 3
- **8-9点**: 優秀（そのまま使用可）
- **6-7点**: 良好（軽微な改善推奨）
- **4-5点**: 要改善（再検証必要）
- **3点以下**: 不適切（使用不可）

## 🚨 エラー検出・修正フロー

### よくあるエラーパターン

#### 1. リンク関連エラー
- **症状**: リンクが切れている、間違ったページ
- **対処**: 公式サイトから正しいURLを再検索
- **予防**: リンク貼り付け直後の動作確認

#### 2. 情報不一致エラー  
- **症状**: 記載内容と引用元の内容が異なる
- **対処**: 引用元を再確認し、正確な情報に修正
- **予防**: 記載前の詳細な照合作業

#### 3. 日付・バージョンエラー
- **症状**: 古い情報、間違ったバージョン番号
- **対処**: 最新の公式情報で確認・修正
- **予防**: 複数ソースでの確認

### 修正作業フロー

1. **エラー検出**
   - セルフチェック実施
   - 品質スコアリング
   - 問題の特定

2. **原因分析**
   - エラーの種類特定
   - 発生原因の調査
   - 影響範囲の確認

3. **修正実施**
   - 正確な情報の再収集
   - 記載内容の修正
   - 関連箇所の確認

4. **再検証**
   - 修正内容の確認
   - 品質スコアの再評価
   - 最終チェック実施

## 🛠️ 支援ツール

### 検証支援スクリプト

#### リンクチェッカー（簡易版）
```javascript
// ブックマークレットとして使用
javascript:(function(){
  const links = document.querySelectorAll('a[href^="http"]');
  let brokenLinks = [];
  
  links.forEach(link => {
    fetch(link.href, {method: 'HEAD'})
      .catch(() => brokenLinks.push(link.href));
  });
  
  setTimeout(() => {
    if(brokenLinks.length > 0) {
      alert('切れたリンクが見つかりました:\n' + brokenLinks.join('\n'));
    } else {
      alert('すべてのリンクが正常です');
    }
  }, 5000);
})();
```

### 品質チェック自動化

#### GitHub Actionsでの基本チェック
```yaml
# .github/workflows/quality-check.yml (将来のPhase 2で実装)
name: News Quality Check
on: [push, pull_request]
jobs:
  quality-check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      - name: Check markdown format
        run: |
          # Markdownの基本的な形式チェック
          find . -name "*.md" -exec markdown-lint {} \;
      - name: Check links
        run: |
          # リンクの基本的なチェック
          find . -name "*.md" -exec markdown-link-check {} \;
```
